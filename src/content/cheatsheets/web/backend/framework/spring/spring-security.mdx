# đ Spring Security Cheatsheet Completo đ

Spring Security es un framework potente y altamente personalizable para autenticaciĂłn y autorizaciĂłn. Se integra perfectamente con aplicaciones Spring y Spring Boot, proporcionando una seguridad robusta para aplicaciones web, APIs REST y microservicios.

## 1. đ Conceptos Clave

* **AutenticaciĂłn**: Proceso de verificar la identidad de un usuario (ÂżQuiĂŠn eres?).
* **AutorizaciĂłn**: Proceso de determinar si un usuario autenticado tiene permiso para realizar una acciĂłn o acceder a un recurso (ÂżQuĂŠ puedes hacer?).
* **`SecurityFilterChain`**: La secuencia de filtros que Spring Security aplica a cada peticiĂłn HTTP para manejar la seguridad.
* **`Authentication` Objeto**: Representa al principal (usuario) actualmente autenticado. Contiene el nombre de usuario, credenciales, y `GrantedAuthorities` (roles/permisos).
* **`GrantedAuthority`**: Una interfaz que representa un permiso otorgado a un principal (usualmente un rol como "ADMIN", "USER").
* **`UserDetailsService`**: Interfaz para cargar los detalles del usuario (nombre de usuario, contraseĂąa, roles) desde una fuente de datos (ej. base de datos, memoria).
* **`UserDetails`**: Interfaz que representa los detalles de un usuario.
* **`PasswordEncoder`**: Interfaz para codificar y verificar contraseĂąas. **ÂĄSiempre usa uno para almacenar contraseĂąas de forma segura!** (ej. BCryptPasswordEncoder).
* **`SecurityContext`**: Almacena el objeto `Authentication` del principal actualmente autenticado.
* **`SecurityContextHolder`**: Un helper que proporciona acceso al `SecurityContext`.

## 3. đ AutenticaciĂłn

### 3.1. AutenticaciĂłn en Memoria (Solo para Desarrollo/Pruebas)

* Define un bean `InMemoryUserDetailsManager` con usuarios y roles.

  ```java
  import org.springframework.security.core.userdetails.User;
  import org.springframework.security.core.userdetails.UserDetails;
  import org.springframework.security.provisioning.InMemoryUserDetailsManager;

  @Bean
  public InMemoryUserDetailsManager userDetailsService(PasswordEncoder passwordEncoder) {
      UserDetails user = User.withUsername("user")
          .password(passwordEncoder.encode("password"))
          .roles("USER")
          .build();
      UserDetails admin = User.withUsername("admin")
          .password(passwordEncoder.encode("adminpass"))
          .roles("ADMIN", "USER")
          .build();
      return new InMemoryUserDetailsManager(user, admin);
  }
  ```

### 3.2. AutenticaciĂłn Basada en Base de Datos (Personalizada)

* Implementa la interfaz `UserDetailsService`.
* Inyecta tu `UserRepository` para cargar usuarios desde la base de datos.
* Tu clase de usuario (`UserEntity`) debe implementar `UserDetails` o convertir a `org.springframework.security.core.userdetails.User`.

  ```java
  import org.springframework.security.core.userdetails.UserDetails;
  import org.springframework.security.core.userdetails.UserDetailsService;
  import org.springframework.security.core.userdetails.UsernameNotFoundException;
  import org.springframework.stereotype.Service;

  @Service
  public class CustomUserDetailsService implements UserDetailsService {
      private final UserRepository userRepository;

      public CustomUserDetailsService(UserRepository userRepository) {
          this.userRepository = userRepository;
      }

      @Override
      public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
          // Asume que UserEntity tiene propiedades como username, password, y roles
          com.example.myapp.model.UserEntity userEntity = userRepository.findByUsername(username)
              .orElseThrow(() -&gt; new UsernameNotFoundException("Usuario no encontrado: " + username));

          // Convierte tu UserEntity a UserDetails de Spring Security
          return User.builder()
              .username(userEntity.getUsername())
              .password(userEntity.getPassword()) // La contraseĂąa ya debe estar codificada en DB
              .roles(userEntity.getRoles().toArray(new String[0])) // Asume getRoles() devuelve List<String>
              .build();
      }
  }
  ```

## 4. đŽ AutorizaciĂłn

### 4.1. AutorizaciĂłn a Nivel de URL (Configurada en `SecurityFilterChain`)

* `authorizeHttpRequests()`: Configura las reglas de autorizaciĂłn para las peticiones HTTP.
* `requestMatchers(patterns).permitAll()`: Permite acceso a todos.
* `requestMatchers(patterns).authenticated()`: Requiere autenticaciĂłn.
* `requestMatchers(patterns).hasRole("ROLE_NAME")`: Requiere un rol especĂ­fico. (Spring aĂąade automĂĄticamente el prefijo "ROLE_" si no lo pones).
* `requestMatchers(patterns).hasAuthority("PERMISSION")`: Requiere una autoridad/permiso especĂ­fico.
* `requestMatchers(patterns).hasAnyRole("ROLE1", "ROLE2")`: Requiere cualquiera de los roles.
* `anyRequest()`: Cualquier otra peticiĂłn no especificada anteriormente.

### 4.2. AutorizaciĂłn a Nivel de MĂŠtodo

* Habilita con `@EnableMethodSecurity` en tu clase `SecurityConfig`.
* **`@PreAuthorize("hasRole('ADMIN')")`**: Se evalĂşa ANTES de ejecutar el mĂŠtodo.

  * `hasRole('ROLE_NAME')`
  * `hasAnyRole('ROLE1', 'ROLE2')`
  * `hasAuthority('PERMISSION')`
  * `hasAnyAuthority('PERM1', 'PERM2')`
  * `isAuthenticated()`
  * `isAnonymous()`
  * `permitAll()`
  * `denyAll()`
  * Puedes usar SpEL (Spring Expression Language) para lĂłgica compleja:
    * `@PreAuthorize("hasRole('ADMIN') or #username == authentication.name")` (Acceso si es ADMIN o si el parĂĄmetro `username` coincide con el nombre del usuario autenticado).
    * `@PreAuthorize("@securityService.canAccess(#id)")` (Llamada a un bean de servicio para lĂłgica de autorizaciĂłn).
* **`@PostAuthorize("returnObject.owner == authentication.name")`**: Se evalĂşa DESPUĂS de ejecutar el mĂŠtodo. Permite verificar el objeto de retorno.
* **`@Secured({"ROLE_ADMIN", "ROLE_USER"})`**: AnotaciĂłn simple para roles (requiere `@EnableGlobalMethodSecurity(securedEnabled = true)` en v5.x o `@EnableMethodSecurity(jsr250Enabled = true)` para `@RolesAllowed`).
* **`@RolesAllowed({"ADMIN", "USER"})`**: EstĂĄndar JSR-250 para roles (similar a `@Secured`, requiere `@EnableMethodSecurity(jsr250Enabled = true)`).

  ```java
  @Service
  @PreAuthorize("isAuthenticated()") // Todos los mĂŠtodos de este servicio requieren autenticaciĂłn
  public class ProductService {

      @PreAuthorize("hasRole('ADMIN')") // Solo admins pueden aĂąadir productos
      public Product addProduct(Product product) { /* ... */ return product; }

      @PreAuthorize("hasAnyRole('ADMIN', 'USER')") // Admins o usuarios pueden ver productos
      public List<Product> getAllProducts() { /* ... */ return List.of(); }

      @PostAuthorize("returnObject.ownerId == authentication.principal.id") // Solo el dueĂąo puede ver su propio producto
      public Product getProductById(Long id) { /* ... */ return new Product(); }
  }
  ```

## 6. đ CaracterĂ­sticas Comunes de Seguridad Web

* **CSRF Protection**: Habilitado por defecto para peticiones POST/PUT/DELETE. Para APIs REST sin sesiones, a menudo se deshabilita (`.csrf(csrf -&gt; csrf.disable())`), pero con precauciĂłn. Para aplicaciones web con formularios, es crucial.
* **CORS Configuration**: Habilita la comunicaciĂłn de origen cruzado.
  ```java

  import org.springframework.web.cors.CorsConfiguration;
  import org.springframework.web.cors.UrlBasedCorsConfigurationSource;
  import org.springframework.web.filter.CorsFilter;

  @Bean
  public CorsFilter corsFilter() {
      UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
      CorsConfiguration config = new CorsConfiguration();
      config.setAllowCredentials(true); // Permitir credenciales (cookies, auth headers)
      config.addAllowedOriginPattern("*"); // Permite todos los orĂ­genes (ÂĄajusta para producciĂłn!)
      config.addAllowedHeader("*"); // Permite todas las cabeceras
      config.addAllowedMethod("*"); // Permite todos los mĂŠtodos HTTP (GET, POST, etc.)
      source.registerCorsConfiguration("/**", config);
      return new CorsFilter(source);
  }
  ```
* **Session Management**: Controla cĂłmo se gestionan las sesiones.
  ```java

  .sessionManagement(session -&gt; session
      .sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED) // STATELESS para APIs sin sesiĂłn
      .maximumSessions(1) // MĂĄximo de sesiones por usuario
      .maxSessionsPreventsLogin(true) // Si se excede, previene nuevo login
      .expiredUrl("/login?expired") // URL de redirecciĂłn si la sesiĂłn expira
  )
  ```

## 8. đĄ Buenas PrĂĄcticas y Consejos

* **Nunca almacenes contraseĂąas en texto plano**: ÂĄSiempre usa `PasswordEncoder`!
* **InyecciĂłn por Constructor**: Prefiere la inyecciĂłn de dependencias por constructor en lugar de `@Autowired` en campos para facilitar las pruebas unitarias y la inmutabilidad.
* **Centraliza la configuraciĂłn de seguridad**: MantĂŠn toda la configuraciĂłn de `HttpSecurity` en una Ăşnica clase `@Configuration`.
* **Roles vs. Permisos (Authorities)**:
  * **Roles** (`ROLE_ADMIN`, `ROLE_USER`): Para agrupar conjuntos de permisos.
  * **Permisos/Authorities** (`READ_PRODUCT`, `WRITE_PRODUCT`): Para control de acceso mĂĄs granular.
  * Usa `hasRole()` para roles y `hasAuthority()` para permisos.
* **Prueba tu seguridad**: AsegĂşrate de que tus reglas de seguridad funcionen como esperas con tests de integraciĂłn (usando `@SpringBootTest` y `MockMvc`).
* **CSRF para Aplicaciones Web**: No deshabilites CSRF si estĂĄs construyendo una aplicaciĂłn web tradicional con formularios, ya que te expone a ataques.
* **AuditorĂ­a de Seguridad**: Considera usar Spring Data Envers o logs detallados para auditar quiĂŠn hace quĂŠ.
